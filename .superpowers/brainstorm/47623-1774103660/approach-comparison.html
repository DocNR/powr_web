Design System Architecture: How should Kinetic Precision live in code?

Three approaches to replacing the current OKLCH theme with Kinetic Precision tokens

A

CSS Custom Properties (Recommended)

Replace the OKLCH variables in globals.css with Kinetic Precision hex values. Add new semantic variables for accent bars, gradients, surfaces. Components consume via var(--primary) — most existing component code stays unchanged.

Pros

  • Smallest diff — swap values, not structure
  • shadcn/ui components auto-inherit new colors
  • Easy to add dark-only theme (no light mode needed for v1)
  • Space Grotesk added as a CSS variable, applied via utility class

Cons

  • Some components may need manual class updates for accent bars, no-border rule
  • OKLCH → hex means losing perceptual uniformity (acceptable for a defined palette)
B

Tailwind Theme Extension

Define Kinetic Precision as a full Tailwind theme with custom colors (text-kinetic-orange), spacing, and component classes. Remove OKLCH entirely. Components updated to use new Tailwind classes.

Pros

  • Tailwind-native — classes are self-documenting
  • Easy to enforce design rules via custom utilities
  • Good IDE autocomplete for design tokens

Cons

  • Larger diff — need to update class names across all components
  • shadcn/ui components need class overrides or re-theming
  • Custom utility classes add Tailwind config complexity
C

Hybrid: CSS Variables + Tailwind Utilities

Keep CSS custom properties as the source of truth (like now), but add Tailwind utility classes that reference them. E.g., bg-surface-card maps to var(--surface-card). Best of both worlds.

Pros

  • CSS variables for theming, Tailwind for ergonomics
  • shadcn/ui inherits via CSS variables automatically
  • Semantic class names like bg-surface-base, text-primary
  • Tailwind v4 @theme inline makes this clean

Cons

  • Slightly more setup than pure CSS variables
  • Two layers to understand (CSS vars + Tailwind mapping)
